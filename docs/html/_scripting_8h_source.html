14// ................................................................................
15// I thought quite a bit about how to do scripting. The main problem is that by
16// allowing users to create their own components and functions its very hard
17// to create a good scripting interface. This problem doesnt occur in other engines
18// as they know all their components upfront and dont expose a lot of internal workings.
19// This approach allows for vast, type safe! customization inside C++, while keeping
20// the configuration manageable. Also this is still universal enough that supporting
21// external scripting is still possible without changing the model.
22// A script here is actually universal for all entities of that type and only exists once!
23// All players have the same logic, all skeletons have the same base logic.
24// Of course their component state can be different and they will behave differently
25// ................................................................................

123{
124 const auto& pos = internal::REGISTRY.get<PositionC>(entity); // Every entity has a position

126 M_ASSERT(script != nullptr, "No Script for this type!");
127 Call<event, Script, entt::registry&, entt::entity, Args...>(script, internal::REGISTRY, entity,
128 std::forward<Args>(arguments)...);
